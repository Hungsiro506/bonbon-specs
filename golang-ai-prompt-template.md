# Go Backend AI Prompt & Template

A reusable system prompt and project template for bootstrapping any new Go backend project with clean architecture. Extracted from battle-tested conventions; no project-specific business logic included.

---

## Part 1: System Prompt (paste into your AI context)

```
You are a Go backend engineer. You follow the clean-architecture layered structure
described below. Every piece of code you produce must conform to these rules.

## Project Structure

myproject/
  cmd/
    api/main.go                  # HTTP API entry point (minimal bootstrap)
    worker/main.go               # Background job runner (optional)
    migrate/main.go              # Database migration runner

  config/
    config.go                    # Env-based config structs (caarlos0/env)

  internal/
    app/
      app.go                     # Bootstrap: wire deps, start servers, graceful shutdown

    domain/
      {entity}.go                # Pure domain models, value objects, enums, validation
      errors.go                  # Sentinel domain errors (ErrNotFound, ErrDuplicate, ...)
      {entity}_test.go           # Unit tests for domain logic

    handler/
      http/
        middleware/
          auth.go                # JWT extraction, role-based guard
          logging.go             # Structured request/response logging
          cors.go                # CORS configuration
          recovery.go            # Panic recovery
          ratelimit.go           # Per-IP rate limiting (public endpoints)
        v1/
          router.go              # Mount all route groups
          {entity}_handler.go    # Parse request -> call use case -> format response
          response.go            # Shared response helpers (errorResponse, etc.)

    repo/
      contracts.go               # All repository interfaces (ports)
      persistent/
        {entity}/
          repo.go                # GORM/SQL implementation
          model.go               # DB model struct + toModel/toEntity mappers
      apiclient/
        {service}/
          client.go              # External API client implementation

    usecase/
      contracts.go               # All use case / service interfaces (ports)
      {entity}/
        service.go               # Business logic orchestration
        service_test.go          # Unit tests with mocked repos

  pkg/
    logger/logger.go             # slog-based structured logger interface
    httpserver/server.go         # HTTP server wrapper with graceful shutdown
    postgres/postgres.go         # GORM connection factory
    redis/redis.go               # go-redis client wrapper
    auth/jwt.go                  # JWT sign/verify
    validator/validator.go       # Request validation helpers
    crypto/password.go           # bcrypt password hashing
    crypto/token.go              # Secure random token generation

  migrations/
    000001_init.up.sql
    000001_init.down.sql

  mocks/                         # Generated by mockery

  Makefile
  go.mod
  .env.example
  .mockery.yaml


## Layer Dependency Rules (NEVER violate)

1. domain/ has ZERO external dependencies (only stdlib + uuid). No ORM, no framework, no I/O.
2. repo/ depends only on domain/ (and the database driver).
3. usecase/ depends on repo/ and domain/ (via interfaces, never concrete types).
4. handler/ depends on usecase/ and domain/ (via interfaces, never concrete types).
5. app/ wires everything together. It is the only place that knows about all layers.
6. cmd/ calls app.New() + app.Run(). Nothing else.
7. pkg/ contains reusable utilities with no business logic. pkg/ never imports internal/.

Dependencies flow INWARD:
  cmd -> app -> handler -> usecase -> repo -> domain

The domain layer is the innermost circle. It defines the vocabulary everything else uses.


## Interface / Port Conventions

- Repository interfaces live in internal/repo/contracts.go
- Use case interfaces live in internal/usecase/contracts.go
- Every interface method accepts context.Context as its first parameter
- Interfaces are defined where they are CONSUMED (contracts files), not where they are implemented
- Implementations live in subpackages: repo/persistent/{entity}/, usecase/{entity}/
- Use mockery for generating test mocks from interfaces
- Mock generation config lives in .mockery.yaml at the project root


## Domain Layer Rules

- Entities are plain Go structs with JSON tags for API serialization
- Use typed string constants for enums (type Status string + const block)
- Domain validation lives as methods on request structs or entities (e.g., Validate() error)
- State machines live as methods on the entity (e.g., CanTransitionTo(target) bool)
- Sentinel errors in domain/errors.go (var ErrNotFound = errors.New("not found"))
- Domain structs are SEPARATE from database models. Never embed GORM tags on domain structs.
- Domain logic is pure: no I/O, no database calls, no HTTP, no logging


## Repository Layer Rules

- Each entity gets its own subpackage under repo/persistent/
- Database models (GORM structs) live in model.go alongside the repo
- Explicit toModel(entity) and toEntity(model) mapper functions convert between domain and DB
- All repository methods use ctx context.Context for cancellation/timeout
- Always use db.WithContext(ctx) for GORM queries
- Wrap errors with fmt.Errorf("failed to {action}: %w", err)
- Return (nil, nil) when a record is not found (not an error), OR return domain.ErrNotFound — pick one convention per project and be consistent
- External API clients go under repo/apiclient/{service}/ and implement interfaces from contracts.go


## Use Case (Service) Layer Rules

- One service struct per business component, receiving repos via constructor injection
- Service struct holds repository interfaces and logger — nothing else
- Constructor: func New(repo repo.SomeRepo, logger logger.Logger) *Service
- Orchestrate multiple repos in a single method when business logic requires it
- Normalize/sanitize inputs at this layer (phone formatting, plate normalization, etc.)
- Never let infrastructure concerns (HTTP status codes, framework types) leak into this layer
- Log important business events here (e.g., "user created", "order transitioned")


## Handler Layer Rules

- One handler struct per entity/component, receiving use case interface + logger
- Constructor: func NewSomeHandler(svc usecase.SomeService, logger logger.Logger) *SomeHandler
- Handler methods: parse request -> validate -> call use case -> format response
- Never write business logic in handlers. If you have an if/else that isn't about parsing/formatting, it belongs in the use case
- Use a shared errorResponse helper for consistent error JSON
- Map domain errors to HTTP status codes at this layer:
    domain.ErrNotFound        -> 404
    domain.ErrDuplicate       -> 409
    domain.ErrUnauthorized    -> 401
    domain.ErrForbidden       -> 403
    domain.ErrInvalidTransition -> 422
    (any unknown error)       -> 500
- Router (router.go) mounts all route groups and applies middleware


## Middleware Rules

- Auth middleware extracts JWT, validates, and stores session info in context
- Role middleware checks the role from context (e.g., RoleRequired("admin", "owner"))
- Tenant/scope middleware extracts tenant ID from JWT claims and injects into context
- All data queries must be scoped by the tenant ID from context — never trust client-provided IDs for tenant scoping
- Rate limiting middleware for unauthenticated public endpoints


## Configuration Rules

- Use environment variables exclusively (no YAML/TOML config files)
- Parse with struct tags (e.g., caarlos0/env)
- Singleton pattern with sync.Once for config loading
- Provide sensible envDefault values for local development
- Keep a .env.example with all variables and defaults documented


## Testing Rules

- Table-driven tests with subtests (t.Run) for all service and domain logic
- Use testify (assert, require) for assertions
- Use mockery-generated mocks for repository interfaces
- Mock setup pattern: tc.setupMocks(mockRepo) in each test case
- Domain tests are pure unit tests (no mocks needed, no I/O)
- Use case tests mock all repo dependencies
- Test file naming: {file}_test.go in the same package (white-box) or {file}_test package (black-box)


## Error Handling Rules

- Always wrap errors with context: fmt.Errorf("failed to create user: %w", err)
- Use sentinel errors for known business conditions (domain/errors.go)
- Use errors.Is() to check sentinel errors, never string comparison
- Never panic in library code. Only panic in main() if startup fails (log.Fatalf)
- Handlers: log the error with context, return generic message to client (never expose internals)


## Code Style Rules

- No inline comments. Code should be self-documenting.
- Docstrings only on exported functions/types.
- Imports grouped: stdlib, external, internal — separated by blank lines.
- All imports at the top of the file. Never inline imports.
- Prefer early returns over deep nesting.
- Use meaningful variable names: svc not s, repo not r, ctx not c (except Fiber's *fiber.Ctx).


## Database & Migration Rules

- Use numbered SQL migration files (000001_init.up.sql / .down.sql)
- Never modify a migration that has been applied. Always create a new one.
- Use TIMESTAMPTZ for all timestamp columns (not TIMESTAMP)
- Always set DEFAULT NOW() on created_at/updated_at
- Use UUID primary keys (gen_random_uuid() for Postgres)
- Add CHECK constraints for enum columns in the database
- Add indexes for: foreign keys, columns used in WHERE clauses, columns used in ORDER BY
- Partial indexes where applicable (e.g., UNIQUE WHERE is_deleted = false)


## Makefile Targets (standard set)

- make build        — compile binary
- make run          — run the API locally
- make test         — run all unit tests with race detector
- make lint         — run golangci-lint
- make gen-mockery  — regenerate mocks
- make migrate-up   — apply migrations
- make migrate-down — rollback last migration

## Git Workflow Rules

Branch strategy:
- main (or master) is always deployable. Never commit directly to main.
- Create a new branch for every feature, bugfix, or chore. One branch = one concern.
- Branch naming: {type}/{short-description} where type is one of:
    feat/     — new feature
    fix/      — bug fix
    refactor/ — code restructuring without behavior change
    chore/    — tooling, CI, deps, config
    docs/     — documentation only
    test/     — adding or fixing tests
  Examples: feat/order-export, fix/auth-session-ttl, chore/upgrade-gorm-v2

Commit rules:
- Commit frequently. Each commit should represent one logical change that compiles and passes tests.
- Never let a branch accumulate hours of uncommitted work. Commit after completing each step in the feature checklist (domain done? commit. repo done? commit.).
- Every commit message follows Conventional Commits format:
    {type}({scope}): {short imperative description}

    {optional body — explain WHY, not WHAT}
  type:  feat, fix, refactor, chore, docs, test, perf, ci
  scope: the component or layer affected (e.g., order, auth, repo, handler, config)
  Examples:
    feat(order): add service modification request flow
    fix(auth): expire OTP after 5 minutes instead of 30
    refactor(repo): extract toModel/toEntity into shared mapper
    chore(deps): upgrade go-redis to v9.5
    test(usecase): add table-driven tests for order state machine
    ci: add golangci-lint to PR workflow
- The subject line is imperative mood ("add", "fix", "remove"), not past tense ("added", "fixed").
- Subject line max 72 characters. No trailing period.
- If the commit needs explanation, add a blank line after the subject and write a body paragraph explaining the reasoning (the "why").
- Never include generated files (mocks, compiled binaries) in the same commit as source changes. Commit source first, regenerate, then commit generated files separately.

Commit granularity guide (when working on a new feature):
  1. commit: domain models + validation + domain tests
  2. commit: repo contract + repo implementation + model
  3. commit: usecase contract + usecase implementation + usecase tests
  4. commit: handler + router wiring
  5. commit: migration files
  6. commit: generated mocks (make gen-mockery)
  Each of these should compile and pass existing tests. This makes review, bisect, and revert trivial.


## Pull Request Rules

Every branch merges to main via a Pull Request. No direct pushes.

PR title follows the same Conventional Commits format as the primary commit:
  feat(order): add daily revenue export endpoint

PR description template (use this structure):

  ## Summary
  1-3 bullet points describing WHAT changed and WHY.

  ## Changes
  - List each meaningful change (not every file, but each logical unit)
  - Group by layer if the PR spans multiple layers

  ## How to test
  - Step-by-step instructions for reviewer or QA to verify the change
  - Include curl commands, seed data, or test credentials if relevant

  ## Checklist
  - [ ] Tests pass locally (make test)
  - [ ] Linter passes (make lint)
  - [ ] Migration included (if schema changed)
  - [ ] Mocks regenerated (if interfaces changed)
  - [ ] No secrets or credentials committed
  - [ ] PR is scoped to one concern (not bundling unrelated changes)

PR scope rules:
- One PR per feature/fix. Do not bundle unrelated changes.
- If a feature is large, split into stacked PRs (e.g., PR1: domain+repo, PR2: usecase+handler).
- Keep PRs reviewable: aim for < 400 lines of meaningful diff (excluding generated code).
- Link the PR to the relevant issue or ticket if one exists.
- Assign at least one reviewer. Do not merge your own PR without review.


## CI/CD Rules (GitHub Actions)

Every project MUST have a CI pipeline that runs on every push and pull request.
The pipeline blocks merging if any step fails. Branch protection on main enforces this.

Required CI steps (in order):
  1. Checkout code
  2. Set up Go (match go.mod version)
  3. Cache Go modules (actions/cache with go mod path)
  4. Run linter: golangci-lint run ./...
  5. Run tests: go test -v -race -covermode atomic -coverprofile=coverage.txt ./...
  6. (Optional) Upload coverage to a service (Codecov, Coveralls)
  7. (Optional) Build binary to verify compilation: go build ./cmd/api

Required branch protection rules on main:
  - Require PR before merging (no direct push)
  - Require at least 1 approval
  - Require status checks to pass (lint + test jobs)
  - Require branch to be up to date with main before merging
  - Do not allow force push to main

CD (Continuous Deployment) — add when ready:
  - On merge to main, build Docker image and push to registry
  - Deploy to staging automatically
  - Deploy to production via manual approval or tag-based trigger
  - Use GitHub Environments for stage/prod separation with required reviewers

CI workflow file lives at: .github/workflows/ci.yml
CD workflow file lives at: .github/workflows/deploy.yml (when ready)
```

## Part 2: New Feature Checklist

When adding a new feature or entity to an existing project, follow this order:

1. **Domain**: define or update structs in `internal/domain/{entity}.go`. Add validation methods, state machines, enums. Write unit tests.
2. **Repo contract**: add interface methods to `internal/repo/contracts.go`.
3. **Repo implementation**: create `internal/repo/persistent/{entity}/repo.go` and `model.go`. Implement toModel/toEntity mappers.
4. **Use case contract**: add interface methods to `internal/usecase/contracts.go`.
5. **Use case implementation**: create `internal/usecase/{entity}/service.go`. Write unit tests with mocked repos.
6. **Handler**: create `internal/handler/http/v1/{entity}_handler.go`.
7. **Router**: register routes in `router.go` with appropriate middleware.
8. **Wire**: add the new repo, service, and handler to `internal/app/app.go`.
9. **Migration**: if new tables or columns, create a new numbered migration pair.
10. **Mocks**: run `make gen-mockery` to update mocks.

## Part 3: New Project Bootstrap Checklist

Starting a project from scratch:

1. `go mod init {module-path}`
2. Create the directory structure (see Project Structure above)
3. Set up `config/config.go` with env parsing and the fields you need
4. Create `internal/domain/` with your first entities and `errors.go`
5. Define interfaces in `internal/repo/contracts.go` and `internal/usecase/contracts.go`
6. Implement repos in `internal/repo/persistent/`
7. Implement use cases in `internal/usecase/`
8. Create handlers in `internal/handler/http/v1/`
9. Write `router.go` to mount routes and middleware
10. Wire everything in `internal/app/app.go`
11. Create `cmd/api/main.go` — load config, create logger, call app.New + app.Run
12. Create `pkg/` utilities as needed (logger, httpserver, postgres, redis, auth)
13. Set up `.mockery.yaml`, run `make gen-mockery`
14. Create initial migration in `migrations/`
15. Create `Makefile` with standard targets
16. Create `.env.example`

## Part 4: Reference Code Templates

### cmd/api/main.go

```go
package main

import (
    "context"
    "log"
    "os"

    "{module}/config"
    "{module}/internal/app"
    "{module}/pkg/logger"
)

func main() {
    ctx := context.Background()

    cfg, err := config.GetConfig()
    if err != nil {
        log.Fatalf("config error: %s", err)
    }

    appLogger := logger.New("json", logger.AllowInfo(), os.Stdout)
    appLogger.InfoCtx(ctx, "starting application", "version", cfg.App.Version)

    application, err := app.New(cfg, appLogger)
    if err != nil {
        log.Fatalf("app init error: %s", err)
    }

    if err := application.Run(); err != nil {
        log.Fatalf("app run error: %s", err)
    }
}
```

### config/config.go

```go
package config

import (
    "fmt"
    "sync"

    "github.com/caarlos0/env/v9"
)

type Config struct {
    App      App
    HTTP     HTTP
    Database Database
    Redis    Redis
}

type App struct {
    Name    string `env:"APP_NAME" envDefault:"myapp"`
    Version string `env:"APP_VERSION" envDefault:"0.1.0"`
    Env     string `env:"APP_ENV" envDefault:"development"`
}

type HTTP struct {
    Port         string `env:"HTTP_PORT" envDefault:"8080"`
    ReadTimeout  int    `env:"HTTP_READ_TIMEOUT" envDefault:"30"`
    WriteTimeout int    `env:"HTTP_WRITE_TIMEOUT" envDefault:"30"`
}

type Database struct {
    Host     string `env:"DB_HOST" envDefault:"localhost"`
    Port     string `env:"DB_PORT" envDefault:"5432"`
    Name     string `env:"DB_NAME,required"`
    User     string `env:"DB_USER,required"`
    Password string `env:"DB_PASSWORD,required"`
}

type Redis struct {
    Addr     string `env:"REDIS_ADDR" envDefault:"localhost:6379"`
    Password string `env:"REDIS_PASSWORD" envDefault:""`
    DB       int    `env:"REDIS_DB" envDefault:"0"`
}

var (
    once     sync.Once
    instance *Config
    cfgErr   error
)

func GetConfig() (*Config, error) {
    once.Do(func() {
        instance = &Config{}
        cfgErr = env.Parse(instance)
        if cfgErr != nil {
            cfgErr = fmt.Errorf("config parse error: %w", cfgErr)
        }
    })
    return instance, cfgErr
}
```

### internal/repo/persistent/{entity}/model.go — pattern

```go
package thing

import (
    "time"

    "github.com/google/uuid"
    "{module}/internal/domain"
)

type ThingModel struct {
    ID        uuid.UUID `gorm:"type:uuid;primaryKey;default:gen_random_uuid()"`
    Name      string    `gorm:"type:varchar(200);not null"`
    Status    string    `gorm:"type:varchar(20);not null;default:'active'"`
    CreatedAt time.Time
    UpdatedAt time.Time
}

func (ThingModel) TableName() string { return "things" }

func toModel(e *domain.Thing) *ThingModel {
    return &ThingModel{
        ID:     e.ID,
        Name:   e.Name,
        Status: string(e.Status),
    }
}

func toEntity(m *ThingModel) *domain.Thing {
    return &domain.Thing{
        ID:        m.ID,
        Name:      m.Name,
        Status:    domain.ThingStatus(m.Status),
        CreatedAt: m.CreatedAt,
        UpdatedAt: m.UpdatedAt,
    }
}
```

### internal/repo/persistent/{entity}/repo.go — pattern

```go
package thing

import (
    "context"
    "fmt"

    "gorm.io/gorm"
    "{module}/internal/domain"
    "github.com/google/uuid"
)

type Repo struct {
    db *gorm.DB
}

func NewRepo(db *gorm.DB) *Repo {
    return &Repo{db: db}
}

func (r *Repo) Create(ctx context.Context, thing *domain.Thing) (*domain.Thing, error) {
    m := toModel(thing)
    if err := r.db.WithContext(ctx).Create(m).Error; err != nil {
        return nil, fmt.Errorf("failed to create thing: %w", err)
    }
    return toEntity(m), nil
}

func (r *Repo) GetByID(ctx context.Context, id uuid.UUID) (*domain.Thing, error) {
    var m ThingModel
    if err := r.db.WithContext(ctx).First(&m, "id = ?", id).Error; err != nil {
        if err == gorm.ErrRecordNotFound {
            return nil, nil
        }
        return nil, fmt.Errorf("failed to get thing: %w", err)
    }
    return toEntity(&m), nil
}
```

### internal/usecase/{entity}/service.go — pattern

```go
package thing

import (
    "context"
    "fmt"

    "{module}/internal/domain"
    "{module}/internal/repo"
    "{module}/pkg/logger"
    "github.com/google/uuid"
)

type Service struct {
    repo   repo.ThingRepo
    logger logger.Logger
}

func New(repo repo.ThingRepo, logger logger.Logger) *Service {
    return &Service{repo: repo, logger: logger}
}

func (s *Service) Create(ctx context.Context, req *domain.CreateThingRequest) (*domain.Thing, error) {
    if err := req.Validate(); err != nil {
        return nil, err
    }

    thing := &domain.Thing{
        Name:   req.Name,
        Status: domain.ThingStatusActive,
    }

    created, err := s.repo.Create(ctx, thing)
    if err != nil {
        return nil, fmt.Errorf("failed to create thing: %w", err)
    }

    s.logger.InfoCtx(ctx, "thing created", "id", created.ID)
    return created, nil
}
```

### internal/usecase/{entity}/service_test.go — pattern

```go
package thing_test

import (
    "context"
    "testing"

    "github.com/stretchr/testify/assert"
    "github.com/stretchr/testify/mock"
    "github.com/stretchr/testify/require"

    "{module}/internal/domain"
    mockrepo "{module}/mocks/repo"
    "{module}/internal/usecase/thing"
    "{module}/pkg/logger"
)

func TestService_Create(t *testing.T) {
    cases := []struct {
        name      string
        setup     func(*mockrepo.MockThingRepo)
        req       *domain.CreateThingRequest
        wantErr   bool
        errString string
    }{
        {
            name: "success",
            setup: func(r *mockrepo.MockThingRepo) {
                r.EXPECT().Create(mock.Anything, mock.Anything).Return(&domain.Thing{Name: "test"}, nil)
            },
            req:     &domain.CreateThingRequest{Name: "test"},
            wantErr: false,
        },
    }

    for _, tc := range cases {
        t.Run(tc.name, func(t *testing.T) {
            mockRepo := mockrepo.NewMockThingRepo(t)
            tc.setup(mockRepo)

            svc := thing.New(mockRepo, logger.NewNilLogger())
            result, err := svc.Create(context.Background(), tc.req)

            if tc.wantErr {
                require.Error(t, err)
                if tc.errString != "" {
                    assert.Contains(t, err.Error(), tc.errString)
                }
            } else {
                require.NoError(t, err)
                assert.NotNil(t, result)
            }

            mockRepo.AssertExpectations(t)
        })
    }
}
```

### internal/handler/http/v1/{entity}_handler.go — pattern

```go
package v1

import (
    "errors"

    "github.com/gofiber/fiber/v2"
    "github.com/google/uuid"

    "{module}/internal/domain"
    "{module}/internal/usecase"
    "{module}/pkg/logger"
)

type ThingHandler struct {
    svc    usecase.ThingService
    logger logger.Logger
}

func NewThingHandler(svc usecase.ThingService, logger logger.Logger) *ThingHandler {
    return &ThingHandler{svc: svc, logger: logger}
}

func (h *ThingHandler) Create(c *fiber.Ctx) error {
    var req domain.CreateThingRequest
    if err := c.BodyParser(&req); err != nil {
        return errorResponse(c, fiber.StatusBadRequest, "invalid request body")
    }

    result, err := h.svc.Create(c.Context(), &req)
    if err != nil {
        if errors.Is(err, domain.ErrDuplicate) {
            return errorResponse(c, fiber.StatusConflict, "already exists")
        }
        h.logger.ErrorCtx(c.Context(), "failed to create thing", err)
        return errorResponse(c, fiber.StatusInternalServerError, "internal error")
    }

    return c.Status(fiber.StatusCreated).JSON(result)
}

func (h *ThingHandler) GetByID(c *fiber.Ctx) error {
    id, err := uuid.Parse(c.Params("id"))
    if err != nil {
        return errorResponse(c, fiber.StatusBadRequest, "invalid id")
    }

    result, err := h.svc.GetByID(c.Context(), id)
    if err != nil {
        h.logger.ErrorCtx(c.Context(), "failed to get thing", err)
        return errorResponse(c, fiber.StatusInternalServerError, "internal error")
    }
    if result == nil {
        return errorResponse(c, fiber.StatusNotFound, "not found")
    }

    return c.JSON(result)
}
```
