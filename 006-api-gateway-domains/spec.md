# Technical Design: API Gateway, Domain Routing & Public Order Authentication

**Date**: 2026-03-14
**Status**: Approved
**Scope**: Domain routing strategy, public order link security, rate limiting, reverse proxy configuration

---

## 1. Problem Statement

BonBon serves three frontend clients and one public-facing order status page. We need:

1. **Domain routing** for production subdomains (gara., car-owner., order., api.)
2. **Public order link security** for Zalo ZNS notification links that require no login but must resist brute-force enumeration
3. **Link expiry** for public order URLs (3-day window for anonymous access)
4. **Rate limiting** to protect the public endpoint from abuse
5. A decision on whether to introduce a dedicated API gateway (Tyk, KrakenD, Kong, etc.)

## 2. Domain Architecture

```
bonbon.com.vn              -> Landing page (out of scope, separate deployment)
gara.bonbon.com.vn         -> garage-web Next.js (:3001) [login required]
car-owner.bonbon.com.vn    -> car-owner-web Next.js (:3002) [login required]
order.bonbon.com.vn        -> car-owner-web Next.js (:3002) [no login, token-based]
api.bonbon.com.vn          -> Go Fiber API (:25502) [mixed auth]
```

### Why order.bonbon.com.vn shares car-owner-web

The `car-owner-web` app already implements `/status/[token]` for public order viewing. Instead of creating a separate service:

- `order.bonbon.com.vn` routes to the same Next.js deployment
- When the car owner eventually logs in via `car-owner.bonbon.com.vn`, they see full order history
- Zero code duplication, single deployment, shared session management

### Reverse Proxy: Caddy (not an API gateway)

**Decision: No dedicated API gateway at this stage.**

Rationale:
- Single Go API binary serving all clients (no multi-service routing needed)
- JWT auth + Redis sessions already implemented in 50 lines of middleware
- Rate limiting achievable with Fiber's built-in limiter (20 lines)
- No external API consumers requiring API keys or throttling
- API gateways (Tyk/KrakenD/Kong) add YAML config complexity, latency, and operational overhead for zero current benefit

**When to revisit**: When we split into multiple Go services, add external partner APIs, or need canary deployments at the route level.

## 3. Public Order Link Security

### 3.1 Current Token Strength

Order tokens are generated by `crypto.GenerateToken(32)` = 64 hex chars = **256 bits of entropy**. Brute-forcing 2^256 is computationally infeasible. For context, UUID v4 has 122 bits; ours has more than double.

### 3.2 Defense-in-Depth Layers

**Layer 1: Token entropy (existing)** -- 256-bit random token, unguessable.

**Layer 2: Rate limiting on /status/:token (new)**
- Per-IP: 30 requests/minute on the public status endpoint
- Returns HTTP 429 with `Retry-After` header

**Layer 3: Token expiry for anonymous access (new)**
- New field: `Order.PublicExpiresAt` = `CreatedAt + 72 hours`
- After expiry, `GET /status/:token` returns HTTP 410 Gone with message: "This link has expired. Log in to view your order history."
- The order data remains permanently in the database
- Authenticated car owners access order history via `/me/vehicles` (no expiry)

**Layer 4 (deferred): HMAC-signed URLs**
- Pre-sign URLs like S3 presigned links: `?exp=UNIX_TS&sig=HMAC(token+exp, SECRET)`
- Server rejects invalid signatures without DB lookup
- Not needed now given 256-bit token + rate limiting + expiry

### 3.3 Zalo ZNS Link Format

```
https://order.bonbon.com.vn/status/{64-char-hex-token}
```

The ZNS template sends this link. The car owner taps it in Zalo, lands on the status page. After 3 days the anonymous link expires, but the car owner can always log in via OTP to see history.

## 4. Implementation Changes

### 4.1 Domain Model

```go
type Order struct {
    // ... existing fields ...
    PublicExpiresAt time.Time `json:"public_expires_at"`
}

func (o *Order) IsPublicAccessExpired(now time.Time) bool {
    return !o.PublicExpiresAt.IsZero() && now.After(o.PublicExpiresAt)
}
```

Default: `PublicExpiresAt = CreatedAt + 72h`, set during `Order.Create`.

### 4.2 New Domain Error

```go
var ErrOrderLinkExpired = errors.New("order link has expired")
```

### 4.3 Rate Limiting Middleware

Using Fiber's built-in `limiter` middleware, scoped to the `/status/:token` route:

```go
func PublicRateLimiter() fiber.Handler {
    return limiter.New(limiter.Config{
        Max:        30,
        Expiration: 1 * time.Minute,
        KeyGenerator: func(c *fiber.Ctx) string {
            return c.IP()
        },
        LimitReached: func(c *fiber.Ctx) error {
            return c.Status(429).JSON(fiber.Map{"error": "too many requests"})
        },
    })
}
```

### 4.4 Handler Change (GetByToken)

```go
func (h *OrderHandler) GetByToken(c *fiber.Ctx) error {
```

### 4.6 Database Migration

```sql
ALTER TABLE orders ADD COLUMN public_expires_at TIMESTAMPTZ;
UPDATE orders SET public_expires_at = created_at + INTERVAL '72 hours' WHERE public_expires_at IS NULL;
ALTER TABLE orders ALTER COLUMN public_expires_at SET NOT NULL;
ALTER TABLE orders ALTER COLUMN public_expires_at SET DEFAULT NOW() + INTERVAL '72 hours';
```

### 4.7 Caddy Reverse Proxy

```
gara.bonbon.com.vn {
    reverse_proxy garage-web:3001
}

car-owner.bonbon.com.vn {
    reverse_proxy car-owner-web:3002
}

order.bonbon.com.vn {
    reverse_proxy car-owner-web:3002
}

api.bonbon.com.vn {
    reverse_proxy api:25502
}
```

Caddy auto-provisions TLS via Let's Encrypt. No manual cert management.

## 5. Security Threat Model

| Threat | Mitigation |
|--------|-----------|
| Token enumeration (brute force) | 256-bit entropy = 2^256 keyspace; rate limit 30/min per IP |
| Leaked link (shared screenshots) | 72-hour expiry on anonymous access |
| DDoS on /status endpoint | Per-IP rate limiting + Caddy can add global rate limiting |
| Session hijacking | JWT in Redis with TTL, HTTPS enforced by Caddy |
| Cross-origin attacks | CORS allowlist for known domains only |
